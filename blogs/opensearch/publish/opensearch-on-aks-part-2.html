OpenSearch on AKS, Part 2: Private Access with Azure Bastion and a Store Search App

Co-authored by Chetan Vaja (Senior Cloud Solution Architect @ Microsoft), Ninad Kapse (Senior Cloud Solution Architect @ Microsoft), Vikas Gautam (Senior Cloud Solution Architect @ Microsoft), and Prafulla Wani (Principal Cloud Solution Architect @ Microsoft).

Summary

Part 1 walked through a starter blueprint for getting OpenSearch running on Azure Kubernetes Service (AKS) with an AKS Azure Verified Modules (AVM) baseline, dedicated manager and data tiers, internal Dashboards access, ClusterIP-only OpenSearch API access, and keyless Azure Blob snapshot storage.

This follow-up intentionally starts where that post stops. Instead of repeating the AKS and Helm deployment walkthrough, Part 2 focuses on the next question platform teams usually ask: how should this look when OpenSearch is treated as a private search platform behind an application, not as an endpoint that users or client applications call directly?

The pattern covered here uses a private AKS API, Azure Bastion native client tunneling for operator access, OpenSearch kept private inside the cluster, and a small store search app that calls OpenSearch from inside AKS. The sample is deliberately simple, but the boundary is the important part: users reach the application; the application reaches OpenSearch.

What changes after Part 1?

Part 1 proves the foundation:

  • OpenSearch manager and data pods are running as StatefulSets.
  • PersistentVolumeClaims are bound to Azure Disk-backed storage.
  • Dashboards is reachable through an internal-only path.
  • The OpenSearch API is validated without making it public.
  • Azure Blob snapshots use workload identity instead of storage account keys.

That is enough for a clean starter deployment. It is not yet the application-facing pattern most teams want to operate. In a real environment, OpenSearch is usually a backing service. Search requests should pass through an application tier that owns request validation, user experience, authorization, throttling, result shaping, and future API gateway or ingress decisions.

This post therefore avoids re-covering the Part 1 deployment steps and adds two things:

  • a private operator access path to a private AKS control plane by using Azure Bastion; and
  • a small store search application that demonstrates how application code should talk to a private OpenSearch service from inside the cluster.

Target architecture for Part 2

The secure flow looks like this:

operator workstation
  -> Azure Bastion native client tunnel
    -> private AKS API
      -> kubectl and Helm operations

browser or test client
  -> store-search app
    -> private OpenSearch ClusterIP service
      -> manager and data StatefulSets with PVC-backed Azure Disks

The access boundaries are explicit:

Surface Recommended exposure Why
AKS API server Private Keeps cluster administration off the public internet.
Operator access Azure Bastion native client tunnel Lets operators run kubectl and Helm from their workstation without a public AKS API or jumpbox VM.
OpenSearch API ClusterIP Keeps the core search API inside Kubernetes.
OpenSearch Dashboards Internal-only operator path Supports administration and exploration without publishing the OpenSearch API.
Store search app Application-owned surface Demonstrates the user-facing pattern while OpenSearch remains private.
Snapshots Azure Blob Storage with workload identity Avoids storage account keys and provides an external recovery target.

The reference implementation assets for this post are in the companion repo: github.com/prwani/oss-data-on-aks.

Why Azure Bastion is useful here

A private AKS API is a good security boundary, but it changes day-to-day operations. If your workstation cannot reach the private endpoint, normal kubectl and Helm commands cannot reach the cluster.

Azure Bastion native client tunneling provides a practical operator path:

  • The AKS API stays private.
  • Operators do not need to publish the AKS control plane.
  • Operators do not need to maintain a jumpbox VM just to run Kubernetes commands.
  • The same Kubernetes and Helm steps from Part 1 can run through the tunnel.

It is important to be precise about what Bastion does and does not do. Bastion gives the operator a private network path to the AKS API. It does not install OpenSearch, create Kubernetes secrets, run Helm charts, seed sample data, or operate the cluster for you. Those remain operator-owned steps, which is usually what platform teams want for a controlled deployment.

Prerequisites

This post assumes you already understand the Part 1 baseline and have the standard local tools installed:

  • Azure CLI, signed in to the target subscription.
  • kubectl.
  • Helm 3.x.
  • git.
  • openssl for local secret generation.
  • An Azure Container Registry (ACR) if you want to build and publish the store search sample image.

For the private access path, you also need Azure Bastion Standard or Premium with native client tunneling enabled. The secure portal template in the repo can create Bastion for the AKS virtual network, or you can use an existing Bastion host in a reachable peered virtual network.

Deploy the secure baseline

If you used the standard baseline from Part 1, the Kubernetes and Helm parts of the OpenSearch deployment are the same. For the private-control-plane pattern, use the secure portal template in the repo:

The secure template creates the private AKS baseline and optional Azure Bastion resources. It intentionally does not use Azure Deployment Scripts. That keeps this flow aligned with environments that disable storage account shared-key access, because the OpenSearch snapshot path is managed identity based.

After the deployment completes, open the deployment outputs and capture these values:

  • deployedClusterName
  • deployedSnapshotStorageAccount
  • snapshotManagedIdentityClientId
  • bastionResourceId
  • getCredentialsCommand
  • aksBastionCommand

The outputs are there so portal-first operators do not need to reverse-engineer resource names before moving to the Kubernetes steps.

Connect to the private AKS API

Get the private cluster credentials first. Use the output command from the deployment or run the command manually:

export RESOURCE_GROUP=<resource-group-name>
export CLUSTER_NAME=<deployedClusterName>
export BASTION_RESOURCE_ID=<bastionResourceId>

az aks get-credentials \
  --admin \
  --resource-group "$RESOURCE_GROUP" \
  --name "$CLUSTER_NAME"

Then open the Bastion tunnel to the private AKS API:

az aks bastion \
  --admin \
  --resource-group "$RESOURCE_GROUP" \
  --name "$CLUSTER_NAME" \
  --bastion "$BASTION_RESOURCE_ID"

Keep that terminal running. In a second terminal, point your kubeconfig at the local tunnel port and verify access:

export BASTION_PORT="$(ps aux | sed -n 's/.*--port \([0-9]*\).*/\1/p' | head -1)"
sed -i "s|server: https://.*|server: https://localhost:${BASTION_PORT}|" "$KUBECONFIG"

export GODEBUG=http2client=0
kubectl get nodes

The GODEBUG=http2client=0 setting can help avoid intermittent HTTP/2 behavior through the local tunnel. Keep the same kubeconfig and environment active while you run the Kubernetes and Helm commands.

Install OpenSearch through the private path

At this point, you do not need a new OpenSearch installation model. Use the namespace, secret, Helm, validation, Dashboards, and snapshot repository steps from Part 1 through the Bastion-backed AKS API connection.

Use the secure deployment outputs for the snapshot values:

export SNAPSHOT_STORAGE_ACCOUNT=<deployedSnapshotStorageAccount>
export SNAPSHOT_IDENTITY_CLIENT_ID=<snapshotManagedIdentityClientId>

Then follow the same release names used in Part 1:

  • opensearch-manager
  • opensearch-data
  • opensearch-dashboards

That reuse is intentional. Part 2 changes the access posture and adds an application tier; it does not introduce a second OpenSearch topology.

Why add an app instead of more API calls?

Part 1 ends with API and Dashboards validation. That is the right place to stop for a baseline post because those checks prove the cluster can index, search, and serve Dashboards without public API exposure.

Part 2 moves one layer up. The store search app shows a more realistic operating model:

browser
  -> store-search service
    -> https://opensearch-manager.opensearch.svc.cluster.local:9200

The sample app indexes a small product catalog and exposes two application endpoints:

Endpoint Purpose
POST /api/seed Creates the product index and loads the demo catalog.
GET /api/search?q=<term>&category=<category> Runs a multi-field OpenSearch query with an optional category filter.

The application uses the internal OpenSearch service name:

https://opensearch-manager.opensearch.svc.cluster.local:9200

That service name only resolves inside the Kubernetes cluster. This is the core pattern: OpenSearch remains a private backing service, while the application becomes the surface you can later expose through an ingress, internal load balancer, API gateway, or private endpoint pattern that fits your organization.

Build and publish the store search image

Clone the repo if you have not already:

git clone https://github.com/prwani/oss-data-on-aks.git
cd oss-data-on-aks

Build the sample into an Azure Container Registry that your AKS cluster can pull from:

export ACR_NAME=<acr-name>
export STORE_SEARCH_IMAGE="${ACR_NAME}.azurecr.io/store-search:0.1.0"

az acr build \
  --registry "$ACR_NAME" \
  --image store-search:0.1.0 \
  workloads/search-analytics/opensearch/samples/store-search

Update the image reference in the sample manifest:

workloads/search-analytics/opensearch/samples/store-search/kubernetes/store-search.yaml

Change:

<acr-name>.azurecr.io/store-search:0.1.0

to the image name you built.

Deploy the app

Create a Kubernetes secret for the app. Use the same OpenSearch admin password that initialized the OpenSearch security index during Part 1.

export OPENSEARCH_ADMIN_PASSWORD='<strong-admin-password>'

kubectl create namespace store-search --dry-run=client -o yaml | kubectl apply -f -

kubectl create secret generic store-search-opensearch \
  --namespace store-search \
  --from-literal=username='admin' \
  --from-literal=password="$OPENSEARCH_ADMIN_PASSWORD" \
  --dry-run=client -o yaml | kubectl apply -f -

Use the same password value for retries. Reapplying this Kubernetes secret with a different value does not rotate the internal OpenSearch admin password after the security index has already been initialized.

Apply the workload manifest and wait for the rollout:

kubectl apply -f workloads/search-analytics/opensearch/samples/store-search/kubernetes/store-search.yaml
kubectl rollout status deploy/store-search -n store-search --timeout=300s

The sample deployment keeps the app service as ClusterIP. That is deliberate for this walkthrough. We can validate the application behavior with port-forward first, then decide separately how the application should be exposed in a real environment.

Test the app

Port-forward the app service:

export GODEBUG=http2client=0
kubectl port-forward svc/store-search 8080:80 -n store-search

In another terminal, seed the product catalog:

curl -s -X POST http://127.0.0.1:8080/api/seed

Then run a search:

curl -s "http://127.0.0.1:8080/api/search?q=running&category=Footwear"

Open the app in a browser:

http://127.0.0.1:8080

The browser talks to the store search app. The app talks to OpenSearch from inside AKS. OpenSearch itself remains private.

What the app demonstrates

The store search sample is intentionally small, but it demonstrates several important behaviors that API-only validation does not show:

  • The app resolves the private Kubernetes DNS name for OpenSearch.
  • The app authenticates to OpenSearch by using a Kubernetes secret instead of embedding credentials in code.
  • The app owns the user-facing API shape.
  • The app can combine full-text search with structured filters.
  • The OpenSearch API does not need a public load balancer.

The sample query uses a multi-field search across product name, brand, and description, plus an optional category filter. That is closer to a real application pattern than calling /_search directly from a user machine.

Operational considerations before publishing the app

The sample is not a production storefront. Before adapting this pattern for a real workload, consider the following areas.

Use least-privilege OpenSearch credentials

The sample uses the admin credential only to keep the walkthrough short. A production app should use an application-specific OpenSearch role and user with only the permissions needed for its index or index pattern.

Expose the app, not OpenSearch

If the app becomes user-facing, expose the application through your chosen ingress, load balancer, private endpoint, or API gateway pattern. Keep OpenSearch private behind the application tier.

Add network controls

Use Kubernetes network policies or your preferred cluster networking controls to limit which namespaces and pods can call the OpenSearch service. A private ClusterIP service is a good start, but it is not the same as workload-level authorization.

Plan credential rotation

Changing a Kubernetes secret is not the same as rotating the credential inside OpenSearch. Document the rotation process for OpenSearch users, Kubernetes secrets, app rollout, and validation.

Keep snapshots and restore drills in the operating model

Azure Disk-backed PVCs protect pod state, but they are not a substitute for OpenSearch snapshots. Keep Blob snapshots, restore testing, retention, and managed identity validation in your day-2 checklist.

Day-2 checks that matter for OpenSearch

A healthy AKS cluster does not automatically mean a healthy OpenSearch cluster. For a durable search platform, monitor both Kubernetes and OpenSearch signals:

Area Signals to watch
Cluster health OpenSearch health status, unassigned shards, node count, and election stability.
Storage PVC binding, disk usage, disk watermarks, shard relocation, and free-space headroom.
Performance Indexing latency, search latency, JVM pressure, garbage collection symptoms, and thread pool saturation.
Recovery Snapshot success, snapshot duration, restore validation, and Blob storage access through workload identity.
Access Dashboards reachability, app reachability, API exposure drift, and Bastion access for operators.

Disk headroom deserves special attention. OpenSearch needs room for segment merges, shard movement, recovery, and temporary imbalance during maintenance. Do not size disks only for raw source data.

required disk ~= raw data * (1 + replicas) * operational overhead + free-space headroom

The exact factor depends on the workload, but the principle is stable: plan for operational movement, not just stored documents.

How Part 1 and Part 2 fit together

The two posts are meant to be read as a sequence:
